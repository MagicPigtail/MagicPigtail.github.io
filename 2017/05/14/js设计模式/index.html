js设计模式

发表于 2017-05-14

设计模式是一条长远的路，但千里之行始于足下，在这里将常用的一些设计模式记下来，之后结合实践时回头看看，相信会有新的感悟和收获。

单例模式

单例就是保证一个类仅有一个实例，并提供一个访问它的全局访问点。单例模式的要点有三个:

  1. 某个类只能有一个实例；
  2. 它必须自行创建这个实例；
  3. 它必须自行向整个系统提供这个实例。
    实现的方法一般是先判断实例存在与否，如果存在直接返回，如果不存在就创建了再返回，这就确保了一个类只有一个实例对象。

策略模式

策略模式即定义一系列的算法，把它们一个一个封装起来。将算法的使用与算法的实现分离开来。
javascript的策略模式很简单，把算法直接定义成函数即可。

策略模式的优点

  1. 有效的避免许多重复的复制粘贴作业。
  2. 开闭原则的完美支持，算法完全独立易于切换、理解、拓展。
  3. 算法复用性强
  4. 使用组合和委托让Validator类拥有执行算法的能力，也是继承的一种轻便替代方式

策略模式的缺点

  1. 增加许多策略类或策略对象。
  2. 将strategy暴露给客户所有实现

策略模式使用场景

  1. 针对同一类型问题的多种处理方式，仅仅是具体行为有差别时；
  2. 需要安全地封装多种同一类型的操作时；
  3. 出现同一抽象类有多个子类，而又需要使用 if-else 或者 switch-case 来选择具体子类时。

观察者及订阅/发布模式

很多地方把观察者模式于订阅/发布模式划上了等号，我阅读了相关的一些文章后，将其理解为这两种模式的思想是一致的，而订阅/发布模式只比观察者模式多了一层消息处理，及发布者发布的消息经过处理后再被订阅者收到。
观察者模式中存在两种角色：观察者（observer）和被观察者(subject)。而事件的发起来源于被观察者的状态改变，subject有一个属性记录所有跟它有观察关系的observer–observerList，在subject的状态变化时，及在它的observerList中遍历每一个observer，然后调用其方法。
相对于订阅/发布模式，观察者模式很直接的绑定被观察者的状态，中间没有经过数据加工。
典型事例：addEventListener。

优点

不管时观察者还是订阅发布模式都降低了事件的发起对象与事件的处理对象之间的耦合性。

缺点

创建订阅者需要消耗一定的时间和内存
